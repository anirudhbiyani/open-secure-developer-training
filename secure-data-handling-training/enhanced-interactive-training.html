🏷️ Data Classification for Developers

As developers, you handle sensitive data constantly. Know your classification levels:

👤 PII
Names, emails, addresses, SSNs, phone numbers—any data that identifies a person
🔑 Authentication Data
Passwords, tokens, API keys, session IDs, JWTs—never log or expose these
💳 Financial Data
Credit cards, bank accounts, transaction details—subject to PCI DSS compliance
🏥 Health Data
Medical records, health information—subject to HIPAA requirements
🔒 Internal/Proprietary Data
Source code, architecture docs, internal APIs—not for public exposure

⚠️ Common Developer Mistakes

  • • Logging PII in debug statements
  • • Using production data in test environments
  • • Returning unnecessary fields in API responses
  • • Storing sensitive data in browser localStorage
  • • Including PII in error messages or stack traces

📋 Developer Data Handling Rules

📊 Data Rules for Every Stage

📥 Collect Only collect the data you need. Minimize PII collection in forms and APIs
💾 Store Encrypt at rest, use parameterized queries, separate PII from analytics data
⚙️ Process Apply data minimization—don't pass full objects when you only need an ID
📋 Log Never log passwords, tokens, credit cards, or PII. Use structured logging with data masking
📤 Return API responses should return only necessary fields. Use DTOs/view models to filter sensitive data
🗑️ Delete Implement data retention policies. Support GDPR right-to-erasure in your data layer

✅ DO's

  • • Use parameterized queries to prevent SQL injection
  • • Mask PII in logs using structured logging frameworks
  • • Use DTOs/view models to control API response fields
  • • Generate synthetic test data instead of using production PII
  • • Store secrets in vaults, not in code or env files

🚫 DON'Ts

  • • Never log full user objects or auth tokens
  • • Avoid SELECT * in production queries
  • • Don't store JWTs or tokens in localStorage
  • • Never hardcode API keys or credentials in source code
  • • Don't include PII in error messages returned to clients

🔑 Golden Rule: Treat every piece of data as if it could end up in a breach report. Minimize, mask, and encrypt!

📜 Data Privacy Compliance for Developers

Privacy regulations like GDPR and CCPA directly impact how you write code. Here's what you must build into your systems:

🗑️ Right to Erasure

Your systems must support deleting a user's data on request. Design for soft-delete with purge capabilities.

📦 Data Portability

Users can request their data. Build export APIs that return user data in a standard format.

✅ Consent Management

Track and respect user consent preferences in your code. Don't process data without proper consent flags.

📉 Data Minimization

Only collect and retain data you need. SELECT specific columns, not SELECT *.

⚠️

🚨 Breach Notification

If your code causes a data breach, your org may have 72 hours to notify authorities. Build monitoring to detect unauthorized access. Log access patterns. Implement alerting for anomalous data access.
