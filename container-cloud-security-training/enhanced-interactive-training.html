🐳 Container Security Fundamentals

📦

Minimal Base Images

Use distroless or Alpine images. Every unnecessary package is an additional attack surface. Never use latest tag—pin specific versions or SHA256 digests.

🏗️

Multi-Stage Builds

Separate build dependencies from runtime. Your production image should only contain what's needed to run—no compilers, no build tools, no source code.

👤

Non-Root Execution

Run containers as non-root users. Add USER nonroot to your Dockerfile. Drop all Linux capabilities and use seccomp profiles.

🔍

Image Scanning

Scan images with Trivy, Snyk, or Grype before deployment. Block images with critical CVEs from reaching production.

🔒

Read-Only Filesystems

Mount container filesystems as read-only. If an attacker gains access, they can't modify the container or persist malware.

🗝️

Secrets in Containers

Never bake secrets into images. Use Kubernetes Secrets, HashiCorp Vault, or cloud provider secret managers injected at runtime.

Found 0 of 6 security issues

🚩 Using ubuntu:latest

Use a specific version like python:3.12-slim or a distroless image. :latest is unpredictable and may include vulnerabilities.

🚩 Installing unnecessary packages

gcc, wget are not needed at runtime. Use multi-stage builds to separate build dependencies.

🚩 Hardcoded DB_PASSWORD

Never put secrets in ENV directives. They're visible in image layers. Use runtime secret injection.

🚩 chmod 777

World-writable permissions are dangerous. Use specific permissions and run as non-root.

🚩 Exposing port 22 (SSH)

Never run SSH in containers. Use kubectl exec or container orchestrator tools for debugging.

🚩 Running as root (no USER directive)

Add USER nonroot:nonroot to run with least privilege.

🔍 Using multi-stage build

✅ This is actually a good practice! Multi-stage builds separate build dependencies from runtime, reducing the attack surface. Not every line is a security issue.

🔍 Specifying WORKDIR /app

✅ Setting a WORKDIR is a neutral/good practice. It avoids ambiguity about where commands run. Not a security issue.

🔍 COPY requirements.txt first for layer caching

✅ This is a good practice! Copying dependency files first optimizes Docker layer caching and doesn't introduce security risks.

✅ Here's the secure version:

# Stage 1: Build FROM python:3.12-slim AS builder COPY requirements.txt . RUN pip install --no-cache-dir -r requirements.txt # Stage 2: Runtime FROM gcr.io/distroless/python3-debian12 COPY --from=builder /usr/local/lib/python3.12 /usr/local/lib/python3.12 COPY --chown=nonroot:nonroot ./app /app USER nonroot:nonroot EXPOSE 8080 CMD ["/app/server.py"]

⎈ Kubernetes Security Essentials

🛡️

Pod Security

Set runAsNonRoot: true, readOnlyRootFilesystem: true, and allowPrivilegeEscalation: false in your pod security context.

🔑

RBAC

Apply least-privilege Role-Based Access Control. Use Roles (namespace-scoped) over ClusterRoles when possible. Audit service account permissions.

🌐

Network Policies

Deny all traffic by default, then allow only required ingress and egress. Prevent lateral movement between pods.

🗝️

Secret Management

Use external secret operators (e.g., External Secrets Operator) instead of base64-encoded Kubernetes Secrets stored in YAML files.

📋

Image Policies

Only allow images from trusted registries. Use admission controllers (OPA Gatekeeper, Kyverno) to enforce image signing and scanning.

📊

Resource Limits

Set CPU and memory limits on all containers. Without limits, a compromised pod can consume all node resources.

☁️ Cloud IAM & Infrastructure as Code Security

🔑 Cloud IAM
🎯

Least Privilege

Start with zero permissions and add only what's needed. Use AWS IAM Access Analyzer or GCP Policy Analyzer to find unused permissions.

⏳

No Long-Lived Keys

Use IAM roles and temporary credentials (STS AssumeRole, workload identity) instead of static access keys.

🧹

Service Account Hygiene

Each service gets its own service account with minimal permissions. Never share or reuse service accounts across services.

🏗️ IaC Security
🔍

Scan Before Deploy

Run Checkov, tfsec, or KICS against Terraform/CloudFormation before applying. Block deployments with critical findings.

🚫

No Hardcoded Values

Use variables and secret references in IaC. Never hardcode IPs, passwords, or account IDs in Terraform files.

📡

Drift Detection

Monitor for manual changes to infrastructure. Drift from your IaC definitions can introduce security gaps.
