📋 OWASP API Security Top 10

Click each item to learn about the most critical API security risks:

1 Broken Object Level Authorization (BOLA) ▼

APIs expose endpoints that handle object IDs. Attackers replace IDs to access other users' data. Always verify the authenticated user owns the requested resource.

2 Broken Authentication ▼

Weak auth mechanisms, missing token validation, or improper credential management allow attackers to impersonate users.

3 Broken Object Property Level Authorization ▼

APIs exposing all object properties (mass assignment) or returning sensitive fields the user shouldn't see.

4 Unrestricted Resource Consumption ▼

No rate limits, no pagination limits, no payload size limits—allowing DoS, brute force, and credential stuffing.

5 Broken Function Level Authorization ▼

Regular users can call admin-only endpoints by guessing or discovering the URL pattern.

6 Unrestricted Access to Sensitive Business Flows ▼

Automated abuse of legitimate business flows like ticket buying, coupon application, or comment posting.

7 Server Side Request Forgery (SSRF) ▼

APIs that fetch URLs provided by users can be tricked into accessing internal services, cloud metadata, or other backends.

8 Security Misconfiguration ▼

Missing security headers, verbose error messages, unnecessary HTTP methods enabled, CORS too permissive.

9 Improper Inventory Management ▼

Deprecated or forgotten API versions still running in production, shadow APIs, undocumented endpoints.

10 Unsafe Consumption of APIs ▼

Blindly trusting data from third-party APIs without validation or sanitization.

🛡️ HTTP Security Headers & CORS

Essential headers every API should implement:

🔒

Content-Security-Policy (CSP)

Prevent XSS by controlling which scripts, styles, and resources can load. Start with default-src 'self' and expand as needed.

🔐

Strict-Transport-Security (HSTS)

Force HTTPS connections. Set max-age=31536000; includeSubDomains; preload to prevent downgrade attacks.

🚫

X-Content-Type-Options

Set nosniff to prevent browsers from MIME-type guessing, which can lead to XSS via uploaded files.

🌐

CORS Configuration

Never use Access-Control-Allow-Origin: * on authenticated endpoints. Whitelist specific trusted origins. Validate the Origin header server-side.

⏱️

Rate Limiting Headers

Return X-RateLimit-Limit, X-RateLimit-Remaining, and Retry-After to communicate limits. Implement per-user and per-IP rate limiting.

🗄️

Cache-Control for Sensitive Data

Set Cache-Control: no-store, private on responses containing user data. Prevent sensitive data from being cached by proxies or browsers.

📊 GraphQL Security Considerations

GraphQL introduces unique attack surfaces beyond REST APIs:

🔎

Introspection in Production

Disable schema introspection in production. It reveals your entire API schema to attackers. Use introspection: false in your server config.

📏

Query Depth & Complexity Limits

Without limits, attackers can craft deeply nested queries that consume all server resources. Set max depth (e.g., 10) and complexity limits.

📦

Batching Attacks

GraphQL allows sending multiple queries in one request. Limit batch size to prevent brute-force attacks (e.g., password guessing) in a single request.

🔁

N+1 Query Prevention

Use DataLoader or similar patterns. Beyond performance, uncontrolled N+1 queries can be weaponized for DoS.

🔑

Field-Level Authorization

Don't rely on removing fields from queries. Enforce authorization at the resolver level for every sensitive field.

✅

Input Validation

Validate all GraphQL input arguments. Custom scalars and input validation libraries help prevent injection through GraphQL variables.
